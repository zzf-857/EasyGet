# EasyGet 智能 Cookie 编排设计

## 目标

让用户在下载公共或已登录内容时尽量不感知 Cookie 配置：EasyGet 先尝试无需登录的公共下载，遇到明确的登录、年龄、机器人验证或站点风控错误后，自动复用本机浏览器登录状态；所有本机浏览器方案均不可用时，才打开一次由 EasyGet 托管的平台登录窗口。单链接解析、播放列表导入、单任务下载和批量下载必须使用同一套认证策略。

“无感”不等于绕过平台登录或安全校验。用户从未登录、会话过期或平台要求验证码时，软件必须展示真实登录页面，完成一次登录后再自动继续任务。

## 当前问题

当前 `YtDlpService` 同时负责站点识别、浏览器探测、Cookie 文件生成、错误分类和重试，认证逻辑散落在视频解析、播放列表解析与下载三个流程中。浏览器读取只覆盖 Chrome/Edge 的 `Default` 配置，并且只有 YouTube 和抖音会尝试浏览器回退。

手动粘贴的无域名 Cookie Header 会被复制到 YouTube、X、Instagram、Bilibili、抖音、TikTok 等多个域名，再作为同一份文件参与所有站点请求。这会造成跨站 Cookie 暴露，也无法判断 Cookie 实际属于哪个平台。设置页仍把粘贴 Cookie 作为主流程，和“用户无感下载”的目标冲突。

批量任务逐条解析元数据；同一平台的多个任务可能重复触发浏览器 Cookie 读取或认证失败。认证状态没有健康记录，后续任务无法优先复用已经成功的策略。

## 方案选择

采用混合模式：匿名快速路径 + 本机浏览器自动探测 + EasyGet 托管登录兜底。

仅自动读取浏览器的方案改动较小，但无法可靠处理 Chromium Cookie 数据库锁定、系统加密策略变化和用户没有登录浏览器的情况。仅使用 EasyGet 托管登录虽然稳定，但会让已经在浏览器登录的用户重复登录。混合模式兼顾默认无感、可靠兜底和最小权限，是本项目的最终方案。

浏览器扩展或 Native Messaging 暂不作为主路径。它会引入安装、浏览器审核和扩展维护成本；仅当未来 Chromium 的应用绑定加密使本机读取长期不可用时，再作为兼容层加入，而不改变本设计的编排接口。

## 架构

新增独立的认证编排模块，`YtDlpService` 只负责构建媒体命令和运行进程。

### `MediaPlatformResolver`

把 URL 解析为稳定的平台标识与域名集合。首批显式覆盖 YouTube、Bilibili、抖音、TikTok、X/Twitter、Instagram、Facebook、快手、小红书、微博、Twitch 和通用 yt-dlp 站点。平台定义包含：可接受域名、登录入口、已知认证错误模式、是否建议匿名优先以及托管会话需要导出的域名。

未知站点仍可使用匿名 yt-dlp；只有错误被可靠判定为认证问题时才进入通用浏览器回退，不能因为普通网络错误无限重试。

### `BrowserProfileDiscoveryService`

只枚举浏览器及配置目录，不直接读取 Cookie 内容。Windows 首批支持 Chrome、Edge、Firefox、Brave、Chromium、Vivaldi 和 Opera，识别默认配置与多个用户配置。发现结果包含 yt-dlp 可用的浏览器标识、配置参数、用户可读名称和最近活动时间，不包含 Cookie 值。

配置排序依次参考：该平台最近成功的配置、当前默认浏览器、最近活动时间、稳定的浏览器优先级。不存在 Cookie 数据库的配置不进入尝试队列。

### `CookieAcquisitionCoordinator`

为视频元数据、播放列表和下载操作生成统一的 `CookieAttemptPlan`，并串联每次进程结果。默认顺序为：

1. 对公共内容使用匿名策略；如果该平台近期持续需要认证且存在最近成功配置，可直接复用成功配置。
2. 使用与当前平台明确关联的旧版手动 Cookie；没有域名的旧版 Header 只能绑定到当前目标平台，禁止复制到其他域。
3. 依次尝试已发现的本机浏览器配置。
4. 全部失败且当前操作允许交互时，通过托管登录请求获取平台会话，然后自动重试。

只有 `CookieFailureClassifier` 判定为认证相关错误时才切换下一策略。网络断开、磁盘错误、格式不可用、用户取消等错误直接返回原始原因，避免不必要的重复下载。

协调器以平台为键维护异步互斥和共享任务。同一批次中只有第一个任务执行浏览器探测或打开登录窗口，其他同平台任务等待并复用结果；不同平台仍可并行。成功策略会用于后续元数据解析与实际下载，避免解析成功后下载阶段重新遍历全部 Cookie。

### `CookieFailureClassifier`

把 yt-dlp 标准错误归一为 `AuthenticationRequired`、`CookieStoreLocked`、`CookieDecryptFailed`、`CookieExpired`、`BotChallenge`、`RateLimited`、`NetworkFailure` 和 `UnrelatedFailure`。分类包含站点专用规则与通用规则，并保留最后一个非 Cookie 根因，最终提示不能被后续浏览器读取错误覆盖。

HTTP 403/412 不自动等同于 Cookie 失效。只有与对应平台、登录提示或已知提取器错误组合出现时才允许认证重试。

### `ManagedLoginSessionService`

使用 WebView2 为各平台建立独立用户数据目录：

`%LocalAppData%/EasyGet/sessions/<platform>/`

登录窗口直接加载平台官网，EasyGet 不接触用户名、密码和验证码。WebView2 会话由 Windows 用户上下文保护。服务在目标域出现可用 Cookie 后关闭窗口并返回会话租约；如果平台继续要求验证码，用户在真实网页内完成即可。

托管登录只在后台策略全部失败后出现一次，并清楚显示目标平台和域名。用户可以取消；取消只终止等待认证的任务，不清除其他平台会话。设置页提供单个平台重新登录、清除会话和清除全部托管会话操作。

### `CookieFileLease`

WebView2 或旧版手动导入需要传给 yt-dlp 时，按平台生成仅包含目标域名的 Netscape 临时文件。文件位于 `%LocalAppData%/EasyGet/temp/cookies/`，仅当前 Windows 用户可访问，进程结束、取消或异常时均通过 `IAsyncDisposable` 删除。应用启动时清理上次崩溃遗留的过期临时文件。

原有永久 `cookies.txt` 退出主流程。原始 Cookie 不写入 `config.json`、日志、下载历史、异常消息或遥测。现有 `CookieContent` 作为兼容迁移来源读取；迁移成功后转为平台作用域数据并清除旧值。

### `CookieHealthStore`

单独保存非敏感元数据：平台、策略类型、浏览器与配置的稳定标识、最后成功时间、连续失败次数和脱敏失败类别。它不保存 Cookie 内容或完整浏览器路径。健康记录用于策略排序、设置页状态和短期失败退避。

## 数据流

### 公共内容

用户粘贴链接后，解析流程使用匿名策略。成功则直接进入队列和下载，用户不会看到登录相关界面。

### 已在浏览器登录的内容

匿名解析返回明确认证错误后，协调器按健康记录选择浏览器配置，并让 yt-dlp 使用 `--cookies-from-browser <browser>:<profile>`。成功配置被记录并直接传递给下载阶段；用户不需要粘贴或导出 Cookie。

### 需要首次登录的内容

所有浏览器配置均失败后，协调器触发一次托管登录。用户在平台官网完成登录，服务导出当前平台所需 Cookie 到临时租约，原任务自动续跑。同平台等待中的批量任务复用该会话。

### 播放列表与批量下载

播放列表导入先完成一次平台认证，再流式产生条目。批量队列只负责排队，不逐条阻塞 UI 等待元数据；后台解析工作池使用单独并发上限。同一平台的认证结果由协调器共享，不会为每个视频重复打开登录窗口。

## UI 与交互

设置页将“Cookie 设置”重构为“智能登录与 Cookie”：

- 默认启用智能模式，主状态显示已检测浏览器数、最近成功平台和是否存在需要重新登录的平台。
- 平台列表使用“可直接下载”“浏览器会话可用”“需要登录”“暂时不可用”四种真实状态，不显示 Cookie 值。
- “登录/重新登录”和“清除会话”按平台操作；高级区域保留手动 Cookie 导入，导入时必须选择目标平台。
- 下载任务增加短暂的“正在获取登录状态”阶段；批量页只显示一个平台级认证提示，不为每个任务堆叠 Toast。
- 浏览器数据库被占用时自动尝试下一个配置；只有没有可用方案时才给出可执行提示。

界面文案不承诺绕过登录，不要求用户关闭所有浏览器作为常规步骤，也不把内部命令行或本地路径暴露给普通用户。

## 错误处理与并发

- 浏览器配置不存在、数据库锁定或 DPAPI 解密失败：记录脱敏健康状态并尝试下一配置。
- Cookie 过期：使该策略降级；若托管会话也过期，只打开一次登录窗口。
- 批量任务认证风暴：平台级异步互斥合并请求，等待任务响应取消令牌。
- 登录窗口被取消：等待该次认证的任务得到统一、可重试的取消结果，其他平台不受影响。
- 下载已产生 `.part` 文件后切换策略：沿用 yt-dlp 续传语义，不创建重复文件。
- 应用崩溃：下次启动清理临时 Cookie 文件；WebView2 持久会话保留，除非用户主动清除。
- 日志：只记录策略类型、平台和失败类别，不记录 Cookie 参数值、浏览器完整配置路径或网页表单内容。

## 兼容与迁移

配置新增版本号和智能 Cookie 开关，默认开启。旧版 Netscape/JSON Cookie 会按其中的真实域名拆分；旧版无域名 Header 不再分发到多个站点，只在用户选择的平台或首次验证成功的平台中使用。迁移前保留现有 `config.backup.json`，迁移成功后清除 `CookieContent` 明文字段。

关闭智能模式时仍可匿名下载公开内容和使用平台作用域的高级手动导入，但不会自动探测浏览器或弹出托管登录。

## 测试

单元测试覆盖：

- URL 到平台/域名的映射与未知站点回退。
- 多浏览器、多配置发现与稳定排序。
- 匿名、旧版平台作用域 Cookie、浏览器、托管登录的策略顺序。
- YouTube、Bilibili、抖音、TikTok、X、Instagram 等站点的认证错误分类。
- 普通网络、格式和磁盘错误不会触发 Cookie 重试。
- 无域名 Header 只写入当前平台域名，绝不跨站复制。
- Cookie 临时文件在成功、失败、取消和异常路径全部删除，日志不包含 Cookie 值。
- 同平台批量任务只触发一次认证，不同平台可以并行。
- 最近成功策略被复用，连续失败策略按规则退避。
- 旧配置迁移和备份行为。
- ViewModel 状态、命令和 XAML 绑定的真实性。

集成测试使用可控的假 yt-dlp 进程验证多次尝试、错误传递和取消。真实站点验收采用手动清单，不把个人账号或 Cookie 放入自动化测试：公共 YouTube/Bilibili、需要登录的 YouTube/X/Instagram、抖音短链、多个浏览器配置、浏览器运行中、会话过期和 20 条同平台批量任务。

每个实施阶段至少执行受影响测试、完整 `dotnet test`、Release 构建和 `git diff --check`。涉及 WebView2 UI 的阶段还要实际渲染设置页与登录窗口，检查 100%、125%、150% 缩放和键盘可达性。

## 完成标准

- 公共内容保持匿名快速下载，不出现登录提示。
- 用户已在任一受支持浏览器登录时，受限内容可以自动选择有效配置并下载。
- 本机浏览器不可用时，同一平台最多出现一次托管登录窗口，登录后等待任务自动继续。
- 同平台批量任务共享认证结果，不重复弹窗或逐任务遍历配置。
- 不再把同一组 Cookie 写入多个无关域名；原始 Cookie 不存在于配置、日志和历史中。
- 设置页能准确展示平台认证健康状态并支持单平台清理。
- 新增测试通过，原有测试无回归，真实站点验收结果被记录。

## 实施分段

1. 抽取平台解析、浏览器配置发现、错误分类和认证计划；修复跨站 Cookie 复制并接入解析/播放列表/下载三个路径。
2. 加入 WebView2 托管登录、临时 Cookie 租约、安全迁移与平台级并发合并。
3. 重构设置页认证状态与交互，并接入批量队列共享认证。
4. 完成真实站点验收、故障修复、文档和发布检查。

每个分段完成并验证后使用中文提交说明并推送到 `origin/main`。
