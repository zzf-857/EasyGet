# EasyGet 抖音模块近 9 小时迭代记录

## 概览

过去约 9 小时，EasyGet 主要围绕抖音模块集成 `douyin-downloader-promax` 能力进行分阶段迭代。工作重点从下载目录策略、并发线程参数、评论采集细项，推进到 sidecar-only 的抖音发现能力基础。

目前前三个功能切片已经完成并提交；发现能力切片处于提交前收尾阶段，重点是打通参数、dry-run 计划、真实运行后的 JSONL 读取与终态输出。本记录不声明发现能力已经完成真实联网验收。

## 过去 9 小时做了什么

### 1. 抖音作者目录策略

提交：`127143c feat: 支持抖音作者目录策略`

完成内容：

- 在 `AppConfig`、设置模型、UI 配置与 sidecar 参数链路中支持 `DouyinAuthorDirectoryMode`。
- sidecar 增加并贯通 `--author-dir` 与 `--no-group-by-mode`。
- 支持控制抖音下载结果是否按作者目录组织。
- 保持配置项与既有设置结构一致，避免为抖音单独引入额外的并行配置体系。

### 2. 抖音并发线程配置

提交：`faca61e feat: 贯通抖音并发线程配置`

完成内容：

- 复用 `AppConfig.ConcurrentFragments` 作为 `douyin-downloader-promax` 的线程参数来源。
- sidecar 增加并贯通 `--thread`。
- 避免新增一套只服务于抖音的并发配置。
- 将已有下载并发语义延伸到抖音 promax 调用侧。

### 3. 抖音评论采集细项

提交：`7e76f68 feat: 支持抖音评论采集细项`

完成内容：

- 增加抖音评论采集相关细项配置：包含二级回复、评论数量上限、评论分页大小。
- UI 与 sidecar 参数链路贯通 `--comment-include-replies`、`--max-comments`、`--comment-page-size`。
- 覆盖评论采集的常用控制项，使下载任务可以更明确地限制评论范围与分页行为。

### 4. sidecar-only 抖音发现能力基础

当前状态：提交前收尾切片，计划与本文档同批提交。

本切片目标是先在 sidecar 层建立抖音发现能力基础，不急于接入完整 UI 页面。

已纳入的能力范围：

- 新增热榜参数 `--hot-board [N]`。
- 新增搜索参数 `--search KEYWORD` 与 `--search-max N`。
- dry-run 模式输出发现计划，便于验证参数解析与执行意图。
- 真实运行时调用 `douyin-downloader-promax` 后，读取 `hot_board/` 或 `search/` 目录下的 JSONL 结果。
- 发现任务完成后输出 `success` 终态，并用 `details.kind=discovery` 区分普通下载任务。
- 测试覆盖参数解析、成功路径与失败路径。

参数决策：

- `--search-max` 必须 `>= 1`。
- 不宣传 `0 = 不限`，因为当前 promax CLI 会将 `0` 解释为默认 `50`，这与“不限”的用户语义不一致。

## 下一步提升计划

- UI 接入发现页：为热榜与搜索增加可操作的前端入口，展示发现结果与任务状态。
- Cookie / 登录态诊断：补充登录态、Cookie 可用性、权限失效等诊断信息，降低用户排错成本。
- 真实网络验收：在明确环境与账号状态的前提下，对热榜与搜索进行真实网络验收。
- 队列 / 批量任务体验：让发现任务与下载任务可以更自然地进入队列、批量运行和追踪。
- 发现结果二次下载：支持从发现结果中选择视频、作者或条目继续创建下载任务。
- 错误分类：区分参数错误、登录态问题、网络失败、promax 执行失败、结果解析失败等类型。
- 文档和用户指南：补充抖音配置项、发现能力、常见问题与限制说明。
- 性能 / 可观测性：增加耗时、结果数量、失败原因、sidecar 调用链路等可观测信息，便于后续优化。
